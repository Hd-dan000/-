# 软件实训教学智能评价系统 —— 赛题核心解读与落地指南

> 本文档整合了关于第十五届“中国软件杯”B组赛题“基于大模型技术的软件实训教学结果检查评价与报表系统”的详细讲解，帮助参赛者精准理解赛题、避免偏题。

---

## 一、赛题一句话定位

**开发一个B/S架构的Web系统，利用大模型技术，对学生在软件实训中提交的各类成果（代码、文档、截图等）进行自动化、智能化的检查和评价，并生成可视化报表。**

- **面向用户**：教师（减负）、学校（质量画像）、学生（即时反馈）
- **核心场景**：高校与企业协同实训，兼顾教学要求与企业标准

---

## 二、绝对不能碰的红线（一票否决）

> **采用B/S架构开发，软件需部署在自主指令系统LoongArch架构+麒麟高级服务器版上运行（不满足该要求视为0分）。**

- 必须为**浏览器-服务器模式**，有Web前端
- 开发与运行环境必须是**龙芯（LoongArch）+银河麒麟高级服务器版**
- 大赛提供龙芯云平台远程环境，请在**该环境中完成开发和部署验证**

---

## 三、五大功能需求拆解

| 功能 | 具体要求 | 实现要点 |
|------|----------|----------|
| **1. 大模型服务 + 可视化界面** | 支持本地部署或云端API调用；具备PC Web或App界面 | 二选一，不能只做后台脚本；推荐前后端分离 |
| **2. 实训成果上传与解析** | 支持Word/PDF、报告、操作截图等多格式上传；**用大模型自动解析内容** | 提取文本、结构、图像信息；存储结构化数据供后续使用 |
| **3. 智能核查** | 实现要求与效果校验、逻辑漏洞识别、步骤完整性核查 | **利用大模型的语义推理能力**，而非简单关键词匹配 |
| **4. 多维度评价管理** | **自定义评价指标+权重分配**；大模型客观评分；**预留教师主观评分入口** | 指标可动态增删改；大模型作为裁判（LLM as Judge）；教师可修改覆盖 |
| **5. 报表生成与导出** | 生成个体评价报告+班级/科目统计报表；支持**Excel/PDF**导出；含**可视化图表** | 图表如雷达图、柱状图；导出文件带图表 |

## 四、学生提交什么？系统对什么打分？

### 4.1 学生提交三类“软件项目产出物”

| 类别 | 文件格式示例 | 具体内容举例（以“图书管理系统”为例） |
|------|-------------|----------------------------------------|
| **源代码** | `.java`, `.py`, `.js`, `.vue`, `.xml`, `.sql` | Controller代码、前端页面、建表脚本 |
| **文档** | `.docx`, `.pdf` | 需求规格说明书、设计文档、测试报告、用户手册 |
| **截图/界面** | `.jpg`, `.png`, `.gif` | 登录界面截图、核心功能操作流程、数据库ER图 |

> 支持**个人提交**和**小组提交**两种模式。

### 4.2 系统针对“企业级交付质量”进行多维度打分

系统**不是笼统打总分**，而是拆解成可自定义的指标维度（教师可设权重）：

| 评价对象 | 典型评价维度 | 大模型如何评判 |
|----------|-------------|----------------|
| **代码** | 代码规范性、代码质量、安全性、复用性 | 检查命名、注释、SQL注入/内存泄漏风险、异常处理等 |
| **文档** | 文档规范性、完整性、可读性 | 检查是否按模板、目录结构、用例描述清晰度 |
| **截图/界面** | 功能实现度、UI一致性 | 多模态大模型识别截图内容是否匹配需求列表 |
| **整体项目** | 步骤完整性、交付标准 | 核查是否缺失部署手册、测试报告等关键步骤 |

### 4.3 系统打分核心逻辑流程

```
教师端：设定评分模型（如：代码40% + 文档30% + 功能30%）
    ↓
学生端：提交 .java + .docx + .png
    ↓
后端：将三类文件分别喂给大模型（含Prompt中嵌入评价标准）
    ↓
大模型输出各项得分及评语（如：代码85分，缺异常处理）
    ↓
系统计算加权总分（85×40% + 90×30% + 80×30% = 85）
    ↓
生成报表：个人雷达图 + 薄弱点分析（“需加强异常处理”）
```

## 五、业务场景深解：“高校-企业协同”

### 5.1 协同内涵
- **共同制定标准**：企业参与实训项目和评价标准的制定
- **引入真实项目**：实训题目来自企业真实需求
- **双导师评价**：学校导师 + 企业导师共同评分
- **课岗对接**：课堂即工位，学习即实战

### 5.2 如何“对接企业标准”落地到系统？
- **评价指标预设**：系统预置企业级规范（如阿里/华为Java开发手册）作为评分模板
- **智能核查带企业视角**：大模型检查代码时，按企业级安全规范审查
- **评价融合校企双方**：系统保留“企业导师”登录入口，可单独打分或复核
- **报表体现岗位胜任力**：雷达图展示学生与企业岗位要求的差距

## 六、教师端导航结构推荐（完全可行）

采用 **“教师端 → 按班级展示 → 进入班级查看实训内容”** 三级结构，完全符合教学场景。

### 进入“实训内容”页面应包含四大板块：

| 板块 | 对应功能 | 内容说明 |
|------|----------|----------|
| **成果提交区** | 功能2 | 展示该生/组提交的代码、文档、截图，带【智能解析】按钮 |
| **多维评价面板** | 功能3 & 4 | 左侧显示大模型各项得分；右侧预留教师/企业导师主观打分输入 |
| **智能核查报告** | 功能3 | 显示检测出的逻辑漏洞、步骤遗漏（红黄绿警示） |
| **导出快捷入口** | 功能5 | 生成个人评价报告PDF或批量导出班级统计Excel |

> **避坑提醒**：进入班级后，最好再增加“按实训科目/项目”分组（如JavaWeb实训、Python爬虫），便于生成“科目实训数据统计报表”。

## 七、必须避开的几个大坑

| 偏题风险 | 正确做法 |
|----------|----------|
| 只做代码评价，忽略文档和截图 | 必须支持**多格式**成果，并分别解析 |
| 大模型只用在后端评分 | 大模型应贯穿**解析 → 核查 → 评价**全过程 |
| 评价指标写死在代码里 | 必须支持**动态自定义**指标和权重 |
| 只有自动评分 | 必须预留**教师/企业导师主观评分**入口 |
| 只用规则或关键词做核查 | 智能核查必须依赖大模型的**语义推理**能力 |
| 报表只展示纯文本数据 | 必须包含**可视化图表**（雷达图、柱状图等） |
| 在x86环境开发完才考虑移植 | 必须在**LoongArch+麒麟**环境中开发和验证 |

## 八、推荐技术路线（供参考）

- **前端**：Vue 3 / React + ECharts（图表）
- **后端**：Python（Flask/Django） / Java（Spring Boot） / Go
- **数据库**：MySQL / PostgreSQL / SQLite
- **大模型**：本地部署（如Qwen） 或 云端API（DeepSeek、通义千问）
- **部署环境**：银河麒麟高级服务器操作系统 V11/V10（LoongArch）

---

## 九、总结一句话

**你的项目必须是一个“能部署在龙芯+麒麟上的Web系统”，让教师可以上传学生的实训成果（代码/文档/截图），系统自动用大模型解析内容、核查问题、按自定义指标打分，最后生成带图表的Excel/PDF报告，同时保留教师和企业的评分入口。每个环节都要有大模型的深度参与，不能只是传统的信息管理系统。**
